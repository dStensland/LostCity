## Data Quality Diagnostic: Placeholder Descriptions in Rec1 Group Descriptions

**Date filed:** 2026-04-16  
**Status:** DB cleaned. Crawler fix required to prevent recurrence.  
**Crawler-dev action required:** YES

---

### Business Impact

The bug caused `events.description` to contain the literal string `"Overall Description of Special Olympics"` — the placeholder heading from Rec1's group-description field — for 9 SOGA-Cobb events (IDs 206465–206473). Event `/atlanta/events/206465` ("2026 Trs Soga-cobb Aquatics") was the production URL reported.

A UI guard was shipped in `web/components/detail/sections/AboutSection.tsx` (regex match on `^overall description of`, `^lorem ipsum`, `^description goes here`, `^\[placeholder\]`) to suppress rendering. The DB has now been cleaned. Both layers are closed, but the crawler can re-introduce the bad data on the next successful crawl if unfixed.

All 9 events are past-dated (February–March 2026) and belong to portal `74c2f211-ee11-453d-8386-ac2861705695`. No future-dated events are currently affected. Because `smart_update_existing_event` runs on each crawl pass, the placeholder was being re-written as recently as 2026-04-15 06:13.

### Data Patterns

**Pattern 1: Rec1 group description is a template heading, not real content.**  
The Rec1 API returns a `group.description` field. For the SOGA programs, that field contains `"Overall Description of Special Olympics"` — which is the boilerplate label text from the Rec1 program-group editing UI, not an actual description. The source left this field with only the section heading. The crawler accepted this value as a valid description without validation.

**Pattern 2: `smart_update_existing_event` re-writes bad data on each crawl.**  
Because the content hash matched (title + venue + date unchanged), the existing record was updated rather than inserted. The update call propagated the same placeholder description, resetting any manual fixes. This is the correct behavior for legitimate updates — but it means a crawler emitting bad data will keep overwriting any DB-side fix until the crawler is patched.

### Root Cause

**File:** `crawlers/sources/_rec1_base.py`  
**Function:** `_build_event_record` (line ~1011)  

```python
# Description: use group description if available, otherwise synthesize
raw_desc = group_description or ""
if raw_desc:
    # Strip HTML
    description = re.sub(r"<[^>]+>", " ", raw_desc)
    description = re.sub(r"\s+", " ", description).strip()
    ...
else:
    # Synthesize a minimal description
    ...
```

The `if raw_desc:` branch fires whenever `group_description` is non-empty — including when the Rec1 source provider left only a section-heading placeholder like `"Overall Description of Special Olympics"`. There is no check that the extracted text is semantically meaningful before writing it to `events.description`.

**The Rec1 API behavior:** When a parks department creates a program group in Rec1 and doesn't fill in the description body, the field contains only the autogenerated section label (`"Overall Description of <Program Name>"`). This is a data-entry artifact, not a real description.

---

### Recommended Fix

**Crawler-dev: modify `_build_event_record` in `crawlers/sources/_rec1_base.py`**

Add a validation step after HTML-stripping the raw description, before writing it to `description`. Reject descriptions that are template headings or pure section labels.

**Suggested implementation:**

```python
# Placeholder/template detection patterns (case-insensitive prefix match)
_PLACEHOLDER_PATTERNS = re.compile(
    r"^(?:"
    r"overall description of\b"  # Rec1 section-heading artifact
    r"|lorem ipsum\b"
    r"|description goes here\b"
    r"|\[placeholder\]"
    r"|coming soon\b"
    r"|no description\b"
    r")",
    re.IGNORECASE,
)

# In _build_event_record, after HTML-stripping raw_desc:
description = re.sub(r"<[^>]+>", " ", raw_desc)
description = re.sub(r"\s+", " ", description).strip()
description = re.sub(r"&amp;", "&", description)
description = re.sub(r"&nbsp;", " ", description)
description = re.sub(r"&lt;", "<", description)
description = re.sub(r"&gt;", ">", description)

# ADDED: reject placeholder/template descriptions — treat as missing
if _PLACEHOLDER_PATTERNS.match(description):
    description = None

if description is None:
    # Fall through to the synthesize branch
    parts = []
    ...
```

Alternatively, restructure the condition:

```python
raw_desc = group_description or ""
description_candidate = None
if raw_desc:
    cleaned = re.sub(r"<[^>]+>", " ", raw_desc)
    cleaned = re.sub(r"\s+", " ", cleaned).strip()
    # ... entity decode ...
    if len(cleaned) > 20 and not _PLACEHOLDER_PATTERNS.match(cleaned):
        description_candidate = cleaned[:600] if len(cleaned) > 600 else cleaned

if description_candidate:
    description = description_candidate
else:
    # Synthesize fallback
    ...
```

The `len > 20` guard also catches near-empty descriptions (e.g. a description that is just whitespace or a single word after stripping HTML).

**This same validation should be applied in any other Rec1 tenant crawler that inherits `_build_event_record` from `_rec1_base.py`.** Since `crawl_tenant()` is the shared entry point, fixing `_rec1_base.py` covers: `cobb_parks_rec.py`, `cobb_family_programs.py`, `dekalb_parks_rec.py`, `gwinnett_parks_rec.py`, `cherokee_county_parks_rec.py`, `forsyth_county_parks_rec.py`, `henry_county_parks_rec.py`, `sandy_springs_parks_rec.py`, and `milton_parks_rec.py`.

### Additional: Validation Rule at Ingestion

Add a validation check in `db/events.py` (or wherever `insert_event` normalizes data before upsert) so that this class of error is caught before writing, regardless of which crawler produced it.

```python
_PLACEHOLDER_DESC_RE = re.compile(
    r"^(?:overall description of|lorem ipsum|description goes here|\[placeholder\]|coming soon|no description)\b",
    re.IGNORECASE,
)

def _validate_description(description: Optional[str]) -> Optional[str]:
    """Return None if description is a known template/placeholder pattern."""
    if not description:
        return None
    stripped = description.strip()
    if _PLACEHOLDER_DESC_RE.match(stripped):
        return None
    if len(stripped) < 10:  # too short to be meaningful
        return None
    return stripped
```

Call this in `insert_event` and `smart_update_existing_event` before writing `description` to the record.
