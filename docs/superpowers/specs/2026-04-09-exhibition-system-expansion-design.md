# Exhibition System Expansion Design

## Problem

LostCity's exhibition system was built for art galleries but the city has museums, zoos, aquariums, botanical gardens, theme parks, and other attractions with notable experiences that drive visits. Current state:

- **Exhibitions table** exists with full infrastructure (schema, dedup, crawlers, API, web pages) but is art-gallery-only and has data quality issues (3x AHC duplication, Kai Lin junk titles, 7 inactive crawlers with working code)
- **`venue_features` table** exists with full pipeline (migration, write path, persistence layer, rendering) but is underutilized — 2,721 records, 0.3% have images, 0% hotel coverage, concentrated in family/arts verticals
- **Exhibitions are Arts-portal-exclusive** — invisible on Atlanta, Family, Adventure, FORTH portals
- **Exhibitions are not searchable** — not a search result type
- **No coverage** of Georgia Aquarium, Zoo Atlanta, Botanical Garden, Six Flags, Children's Museum, or any major attraction venue

## Design Decisions

1. **Evolve `venue_features`, don't create a new table.** The table exists, is wired end-to-end, and covers ~70% of what was proposed as `place_features`. Add missing columns via ALTER TABLE migration.

2. **Expand `exhibitions` for non-art venues.** New `exhibition_type` values handle seasonal events, special exhibits, and temporary attractions. Same table, same pipeline, same API.

3. **Two entity types, clear boundary:**
   - `venue_features` = what the place IS (permanent or seasonal attractions, physical installations)
   - `exhibitions` = curated time-boxed content with editorial identity (shows, seasonal events, limited-run experiences)
   - Decision rule: does it have a start/end date and would draw someone to visit specifically for it? → exhibition. Is it a persistent reason the place is worth going to? → venue feature.
   - Edge case — seasonal recurring experiences (e.g., "Garden Lights" returns every winter): these are `exhibitions` with `exhibition_type = 'seasonal'`, not venue features with `is_seasonal = true`. The feature `is_seasonal` flag is for permanent attractions that close part of the year (e.g., a water play area closed in winter). The exhibition `seasonal` type is for programmed experiences with a name, dates, and marketing identity that recur annually as distinct runs.

4. **Unified venue research pipeline** built as crawlers within existing `main.py`, not a separate system. One crawl pass per venue produces both features and exhibitions.

5. **Cross-portal exhibition visibility.** Non-art exhibition types surface on relevant portals. Each portal's feed shell owns its section with bespoke title and card variant — no config table, just code constants.

6. **Features enrich place pages only (for now).** No standalone feature detail pages. Revisit if search/SEO signal warrants it.

7. **Fix existing data quality before expanding.** Pipeline issues (duplication, junk titles, inactive crawlers) are preconditions, not separate work.

## Schema Changes

### `venue_features` ALTER TABLE (additive columns)

| Column | Type | Notes |
|---|---|---|
| `source_id` | INTEGER FK → sources | Nullable. Tracks which crawler/research pass produced this |
| `portal_id` | UUID FK → portals | Nullable. Federation scoping |
| `admission_type` | TEXT | Nullable. Values: `free`, `ticketed`, `included`, `donation`, `suggested` |
| `admission_url` | TEXT | Link to buy tickets / reserve |
| `source_url` | TEXT | Provenance — where the data came from |
| `tags` | TEXT[] | Flexible tagging for discovery |
| `metadata` | JSONB DEFAULT '{}' | `content_hash` for dedup, `last_verified_at` for staleness |
| `created_at` | TIMESTAMPTZ DEFAULT now() | Use `IF NOT EXISTS` — may already be present |
| `updated_at` | TIMESTAMPTZ DEFAULT now() | Use `IF NOT EXISTS` — may already be present |

Add `update_updated_at_column()` trigger on `venue_features` if not already present — without it, `updated_at` stays at insertion time forever and staleness tracking via `metadata.last_verified_at` won't work.

Existing columns preserved as-is: `title`, `feature_type`, `description`, `image_url`, `is_seasonal`, `start_date`, `end_date`, `price_note`, `is_free`, `sort_order`, `is_active`.

### `exhibitions` ALTER TABLE

| Change | Detail |
|---|---|
| Add `related_feature_id` | **BIGINT** FK → `venue_features(id)`, nullable. Links time-boxed exhibition to the permanent feature it's within. Must be BIGINT (not UUID) because `venue_features.id` is `BIGINT GENERATED BY DEFAULT AS IDENTITY`. |
| Add index | `idx_exhibitions_related_feature ON exhibitions(related_feature_id) WHERE related_feature_id IS NOT NULL` |
| Expand `exhibition_type` | Add values: `seasonal`, `special-exhibit`, `attraction`. Requires DROP + re-ADD of the CHECK constraint (cannot append to existing CHECK). Verify constraint name in live DB before writing migration. |
| Expand `admission_type` | Add value: `included` (comes with general admission). Same DROP + re-ADD pattern for CHECK constraint. |

## Data Pipeline

### Unified venue research crawlers

Each target venue gets a source registration and crawler file in the existing pipeline:

1. **One crawler per venue** (or venue group for similar sites). Registered with `crawl_frequency` — `weekly` for time-boxed exhibitions, `monthly` for feature-only venues. (The pipeline doesn't support `quarterly`; `monthly` is the longest supported cadence and is sufficient for permanent features.)

2. **Each crawl produces two entity types** in a single pass:
   - `venue_features` — permanent/notable attractions
   - `exhibitions` — current time-boxed content
   
   Both lanes already exist in `entity_lanes.py`.

3. **LLM extraction** for marketing-heavy, unstructured sites (Six Flags, Botanical Garden). Prompt instructs: "Extract permanent notable attractions AND current limited-time experiences separately." Same pattern as `exhibition_hub.py`.

4. **Quality gates at ingestion:**
   - Features require: title, description (>=100 chars), valid `feature_type`
   - Image extraction required when source page has one
   - Exhibitions follow existing `insert_exhibition()` validation
   - `feature_type` validated against known values, unknown defaults to `attraction` with warning

### Refresh cadence

| Content type | Frequency | Mechanism |
|---|---|---|
| Permanent features | Monthly | `crawl_frequency = 'monthly'` on source (longest supported cadence) |
| Seasonal features (`is_seasonal = true`, `end_date` within 60 days) | Weekly | Bumped via staleness check |
| Time-boxed exhibitions | Weekly | Normal `crawl_frequency` |
| Feature staleness | Tracked via `metadata.last_verified_at` | Flagged if >120 days stale |

### Venue page acceptance criteria

A seeded venue is "done" when its place detail page shows:
- All notable permanent features with images and >=100 char descriptions
- Any current time-boxed exhibitions with dates, description, and image
- No empty sections rendered (hide "What's Here" / "On View" if no data)
- Renders cleanly at 375px mobile width

## Product Surface

### Cross-portal exhibition visibility

New exhibition types surface on non-Arts portals. Each portal's feed shell owns its own section — bespoke title, bespoke card variant, type filter as a code constant. No config table, no shared section component with title props.

| Portal | Exhibition types | Section title | Card component |
|---|---|---|---|
| Atlanta (base) | All types | "What's On Now" | Generic `ExhibitionRowCard` (new) |
| Family | `seasonal`, `special-exhibit`, `attraction` | "On Now for Kids" | Generic `ExhibitionRowCard` |
| Adventure | `seasonal`, `attraction` | "What's On Now" | Generic `ExhibitionRowCard` |
| FORTH (hotel) | All types | "Special Exhibits & Seasonal Experiences" | Generic `ExhibitionRowCard` |
| Arts | `solo`, `group`, `installation`, `retrospective`, `popup` | "What's Showing" (existing, unchanged) | Existing `ExhibitionCard` (arts-specific, unchanged) |

The existing `ExhibitionCard` in `ArtsFeedShell` hardcodes `category: "art"` and uses arts-specific accent colors. Non-arts portals must use a new generic `ExhibitionRowCard` that derives category from venue type, not from a hardcoded art default.

### Exhibitions in search

Add `exhibition` as a search result type in unified search. Results show: title, venue name, date range, admission type, thumbnail. Searching "aquarium" returns Georgia Aquarium (place) + any current special exhibit (exhibition).

### On-venue exhibition links

"On View" cards on venue detail pages link to `/{portal}/exhibitions/{slug}` (internal detail page) instead of `source_url`. External source link stays available on the exhibition detail page.

### Portal-aware exhibitions page

Existing `/{portal}/exhibitions` route works for any portal. Requires fixes:

- **Remove hardcoded `ArtsSecondaryNav`** — currently renders arts-specific nav chrome regardless of portal. Must be portal-conditional: arts portal renders `ArtsSecondaryNav`, others render their own portal shell nav or none.
- **Portal-aware page copy** — "Currently showing across Atlanta's galleries and museums" is hardcoded and wrong for non-arts portals. Derive from portal context.
- **Page title adapts** — "What's Showing" on Arts, "What's On" elsewhere.
- **Type-based filtering on Atlanta base portal** — the listing page shows all types unfiltered. Add type tabs or filter chips so art exhibitions and Fright Fest don't sit side-by-side without differentiation.

### Exhibition detail page — conditional rendering by type

The current detail page (`/{portal}/exhibitions/[slug]/page.tsx`) is structurally art-world: Playfair Display italic titles, curator credit section, "View on gallery website" source link, `medium` metadata field. None of this maps to a Six Flags Fright Fest or Aquarium special exhibit.

**Art types** (`solo`, `group`, `installation`, `retrospective`, `popup`, `permanent`): Keep current arts template — Playfair Display, artist/curator credits, medium field, "View on gallery website."

**Non-art types** (`seasonal`, `special-exhibit`, `attraction`): Different rendering path:
- Title in platform standard (cream/semibold), not Playfair italic
- No artist/curator credit section
- No `medium` metadata field
- Source link says "Visit website" not "View on gallery website"
- Type badge renders contextual label ("Seasonal Event", "Special Exhibit", "Limited-Time Attraction") not "// exhibition"
- Closing countdown calibrated to type — 30-day window is gallery-appropriate but meaningless for a 6-month special exhibit. Use 14 days for `seasonal`, 30 days for `special-exhibit`, 7 days for `attraction`.

### Venue detail page — accumulation management

High-content venues (Georgia Aquarium) could render 10+ features + seasonal features + current exhibitions. Before shipping "On View" cards alongside features:

- **Audit at 375px** using Georgia Aquarium as the stress test case
- **Consider merging** features and current exhibitions into a single "What's Here" section with type badges, rather than two separate sections
- **Cap if needed** — show at most 3 "On View" exhibition cards, link to full listing
- The existing `PlaceFeaturesSection` already has `hideType` logic and grid/list switching — extend it rather than creating a parallel section

## Pipeline Code Changes

These files need updating alongside schema migrations — without them, new columns are silently dropped at ingestion or missing from API responses.

### Crawler layer (Phase 2)
- **`crawlers/db/exhibitions.py`** — `_EXHIBITION_COLUMNS` whitelist (line ~94-98) must add `related_feature_id`. Without this, crawlers that emit the FK will silently lose the link. Same set used by `update_exhibition()`.
- **`crawlers/db/places.py`** — `upsert_venue_feature()` (line ~1021-1036) builds a fixed `row` dict. Must add `source_id`, `portal_id`, `admission_type`, `admission_url`, `source_url`, `tags`, `metadata` to the column set.
- **`crawlers/entity_persistence.py`** — `persist_typed_entity_envelope()` calls `upsert_venue_feature` and discards the returned ID (line ~99). For `related_feature_id` linking, exhibitions need to reference feature IDs. Add a feature-ID-by-slug lookup or accumulate returned IDs during persistence so exhibitions can reference them.

### Web layer (Phase 2)
- **`web/lib/place-features.ts`** — `PlaceFeature` TypeScript type must add: `source_id`, `portal_id`, `admission_type`, `admission_url`, `source_url`, `tags`, `metadata`. Also add `space` → `amenity` mapping or add `space` to `FeatureType` union.
- **`web/lib/types/exhibitions.ts`** — `ExhibitionType` union must add `seasonal`, `special-exhibit`, `attraction`. `AdmissionType` must add `included`. Add `related_feature_id: number | null` to `Exhibition` interface. Update `EXHIBITION_TYPE_LABELS` and `ADMISSION_TYPE_LABELS` maps.
- **`web/app/api/exhibitions/route.ts`** — verify the `type` query parameter filter works with new type values (it should — it's a passthrough to Supabase `.eq()`).

## Implementation Sequence

### Phase 1: Pipeline Quality (prerequisite, shared with P4)
All six data quality fixes above. No schema changes, no new features — just fixing what's broken. This work benefits both the Arts portal (P4) and this expansion. Can proceed immediately.

### Phase 2: Schema Evolution (gate: P4 exhibitions page is live)
Do not alter the exhibitions table while P4 is mid-build — migration conflicts and schema drift will cause problems. Wait until P4's exhibitions page is live and working, then:
- `venue_features` ALTER TABLE migration (additive columns, `IF NOT EXISTS` guards, `updated_at` trigger)
- `exhibitions` ALTER TABLE (`related_feature_id` as BIGINT FK, DROP + re-ADD CHECK constraints for expanded enums)
- Update pipeline code: `_EXHIBITION_COLUMNS`, `upsert_venue_feature`, `PlaceFeature` type, `ExhibitionType` union
- Add `exhibition` to unified search index

### Phase 3: Venue Research Pipeline
- Build LLM-extraction crawler template for attraction-type venues
- **Launch cohort (7 venues):** Georgia Aquarium, Zoo Atlanta, Botanical Garden, High Museum, Fernbank, Children's Museum, World of Coca-Cola. Hard quality gate: each venue must meet acceptance criteria before moving on.
- **Expansion cohort (after launch quality verified):** Six Flags, Stone Mountain, Center for Civil & Human Rights, College Football HoF, Center for Puppetry Arts, Legoland, Medieval Times, Delta Flight Museum, plus gallery activations.
- Each seed produces both `venue_features` and `exhibitions`

### Phase 4: Product Surface
- Exhibition detail page conditional rendering (art vs. non-art types)
- Cross-portal exhibition feed sections (bespoke per portal shell, generic `ExhibitionRowCard`)
- Exhibitions in unified search results
- Fix `/{portal}/exhibitions` page: remove hardcoded `ArtsSecondaryNav`, portal-aware copy/title
- "On View" links point to internal detail pages
- Venue detail page accumulation audit at 375px

### Phase 5: Ongoing Maintenance
- Monthly re-crawl for permanent features (`crawl_frequency = 'monthly'`)
- Weekly for seasonal features approaching `end_date`
- Weekly crawls for time-boxed exhibitions at active venues

## B2B / FORTH Angle

The launch cohort of 7 venues (Aquarium, Zoo, Botanical Garden, High Museum, Fernbank, Children's Museum, Coca-Cola) are also the top venues a hotel concierge recommends to guests. Properly seeded exhibition data for these venues is a direct FORTH sales asset — a hotel GM seeing "Lantern Festival at Georgia Aquarium through November" in their concierge feed is a concrete demo of platform value. This is not just product improvement; it's B2B pipeline fuel.

## Out of Scope

- Standalone detail pages for venue features (revisit with user signal)
- Structured pricing on features (admission_note / price_note handles the variety)
- Artist identity resolution across exhibitions (existing gap, separate project)
- Feature-to-feature relationships (e.g., "Ocean Voyager is part of Georgia Aquarium's Marine Galleries")
