Expert Review Synthesis

Three reviewers, distilled into what actually needs to change in the spec.

Critical — Must Address

Design + Architect Mobile scroll depth buries type-priority content
On mobile, sidebar stacks above content: hero (~211px) + identity zone (~200px) + actions (~80px) = ~500px of chrome before the first section renders. The entire point of type-priority manifests (cinema leads with showtimes) is defeated on mobile because showtimes are below the fold.
Resolution: Add mobile-specific layout behavior to the spec. Options: compact identity strip on mobile, or inject type-priority section into sidebar on mobile. Film poster (2:3) is the worst case at 563px — needs max-height constraint on mobile.
Architect EntityData union type is a type-safety illusion
The 5 API responses have incompatible shapes (event.venue.slug vs spot.slug, eventArtists vs programs). Every section will immediately cast away the union. No compile-time prevention of putting `lineup` in a place manifest.
Resolution: Use discriminated union with entityType as discriminant. Add allowedEntityTypes to SectionModule for runtime guards. Define SectionId as a string literal union for manifest type-checking.
Architect resolveConnections is a god function
Single function handling 5 entity types with completely different data shapes = 200+ line if/else branch. Untestable, unmaintainable.
Resolution: Five per-type resolver functions (resolveEventConnections, resolvePlaceConnections, etc.) returning shared ConnectionRow[]. Orchestrator calls the right one.
Strategist ATLFF data graph isn't wired (festival_id = null)
ConnectionsSection is the showpiece feature. But ATLFF events currently have festival_id = null in the DB. If the crawler fix doesn't land before April 24, the gold festival connections render nothing. The architecture enables it; the crawler fix actualizes it.
Resolution: Add crawler fix as explicit P0 dependency in the spec. festival_id linkage must land regardless of rearchitecture timing.
Design Event manifest is undefined
Section 3.3 defines manifests for 6 place subtypes but never specifies the event manifest, which is the most visited entity type. The ordering of lineup vs socialProof vs showSignals vs gettingThere is a UX decision, not an implementation detail.
Resolution: Add explicit event, series (film + non-film), festival, and org manifests to the spec.

Important — Should Address

Missing manifests have gaps
Cinema omits eventsAtVenue (Q&As, midnight screenings). Museum omits socialProof. Bar omits accolades. Music venue omits specials. Trait filtering handles empties — no penalty for including them.
Empty/thin state unspecified
When only 1-2 sections render, the page is 90% chrome. Need a minimum-viable page state per entity type and a thin-state comp in the Pencil list.
Loading/error states not specified
Current views have bespoke skeletons. A generic skeleton is a regression. Needs explicit treatment — shared skeleton with type-specific slots, or per-type skeletons.
ScopedStyles / accent color plumbing unspecified
Every current view uses createCssVarClass for dynamic accent colors. Spec says sections get accentColor string but doesn't specify how it's applied. DetailLayout should render ScopedStyles once.
SocialProof vs Connections boundary unclear
Matrix shows socialProof for Event/Place. But connections also includes "friends going." Where does friend attendance render — both sections?
SectionWrapper count badge source undefined
SectionWrapper renders SectionHeader with count badge, but SectionModule interface has no count field. Who provides the count?
Connection context lines should be relationship-first, not quantity-first
"High Museum — 14 upcoming events" is noise. "High Museum — hosting 3 ATLFF screenings this week" is a reason to tap. Context should answer "why now?" not "how many?"
Logged-out state not addressed
ATLFF marketing push brings new users from Instagram. Social proof sections gate on hasSocialData — renders nothing for logged-out users. Need explicit logged-out treatment.
Place subtype mapping unspecified
getPlaceManifest uses "Cinema", "Restaurant" but spot_type values are "movie_theater", "restaurant". The mapping needs to be explicit for 70+ types.

Strategic Callouts

Comps gate: design what's novel, build the rest from spec
Strategist argues 21 comps before code adds 1-2 weeks for mostly-extraction work. Counter-proposal: comp ConnectionsSection + FestivalDetailView + thin-state (genuinely novel), build the other 17 section modules directly from the prose spec. Cuts design gate from 2 weeks to 3-4 days.
B2B value is real
All three reviewers agree: composable manifests per portal is the right architecture for multi-vertical detail pages. Hotel portal leads with directions. Hospital portal leads with programs. This is the pitch.
ConnectionsSection is differentiated — but only with data
No competitor does full cross-entity graph surfacing in a local-temporal context. But it requires populated FKs (festival_id, producer_id, venue series). Architecture enables it; data actualizes it.
